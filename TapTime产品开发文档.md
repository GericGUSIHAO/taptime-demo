# TapTime 产品开发文档

> **产品名称**：TapTime（时刻）
> **产品形态**：移动端 App（Demo 用交互式网页展示）
> **一句话定位**：轻点一下，记录时间。知道自己怎么过的，成为想成为的人。

---

## 一、产品概述

### 1.1 产品定位

TapTime 是一个**基于真实时间分配的习惯养成工具**。

不是又一个打卡 App，不是又一个时间记录 App。
它解决的核心问题是：**大多数习惯工具假设用户每天都有固定时间和意志力，但现实生活根本不是这样。**

### 1.2 核心差异化

| 传统工具 | TapTime |
|---|---|
| 先定习惯，再找时间 | 先看时间，再安排习惯 |
| 每天都要打卡，断了就焦虑 | 习惯养成后反向打卡，断了不焦虑 |
| 记录很详细，维护成本高 | 点一下就完成，15 秒内搞定 |
| 功能堆砌，大而全 | 三个核心功能，克制到极致 |

### 1.3 核心理念

> **不是靠意志力坚持，是靠时间和系统。**
> 打卡不是目的，养成习惯才是目的。
> 让工具服务人，而不是让人伺候工具。
> 记录本身就是目的 —— 没有连续打卡压力，没有断签焦虑。

### 1.4 四大痛点

| 痛点 | 描述 | 本质问题 |
|---|---|---|
| **打卡永远会断** | 坚持 30 天，第 31 天忘了，就再也不想打开 | 打卡靠"每天记得"，不是"习惯真养成了" |
| **时间记录太繁琐** | 每件事都要记开始时间、结束时间、事情名称 | 记录本身变成了负担 |
| **计划不可行** | "每天健身 1 小时"——但根本没有那么多时间 | 习惯没绑定真实时间 |
| **几年下来原地踏步** | 学了很多但生活没有变化 | 只有记录没有方向，缺少"理想"的对比 |

## 二、目标用户

### 2.1 用户画像

| 维度 | 描述 |
|---|---|
| **人群** | 25-35 岁职场人，想改变但没方法 |
| **特征** | 学了很多方法论但生活没变化；下载过很多效率 App 但都用不久 |
| **痛点** | 打卡会断、时间记录太繁琐、Flag 年年立年年倒 |
| **心理** | 不是不自律，是那些工具本身就设计错了 |

### 2.2 使用场景

| 时间 | 行为 | 耗时 |
|---|---|---|
| **早上** | 打开 App，看一眼今天有哪些事 | 约 30 秒 |
| **白天** | 做完一件事，顺手点一下记录 | 约 3 秒 |
| **晚上** | 点一下记录今天做了什么，看看数据 | 约 1 分钟 |

**每天 1-2 次，每次不超过 2 分钟。**

## 三、功能定义

### 3.1 MVP 功能范围

| 功能 | 说明 | 优先级 |
|---|---|---|
| **时间块记录** | 打开 APP 直接看到类别卡片，点击卡片 1 次=15 分钟，短时间内多次点击合并为一条记录（n×15 分钟） | ✅ P0 核心 |
| **今日视图** | 显示今天所有记录的时间块列表，按时间倒序排列，支持删除 | ✅ P0 核心 |
| **今日合计** | 实时计算并显示今日总记录时长，数字有递增动画 | ✅ P0 核心 |
| **本周时间块视图** | 今日时间块可视化排列，每个块按类别着色，显示时间范围 | ✅ P0 核心 |
| **本周分布概览** | 本周各活动类别时长分布，7 天圆点日历 | ✅ P0 核心 |
| **理想周排程** | 按天设置时间块（如周一 8:30-18:00 工作），不同活动不同颜色，空白处显示可安排时间 | ✅ P0 核心 |
| **底部 Tab 导航** | 今日/本周/理想三个 Tab 切换，带动画效果 | ✅ P0 核心 |
| **数据持久化** | 使用 localStorage 存储，刷新页面后数据不丢失 | ✅ P0 核心 |
| **预设数据** | 首次打开有示例数据，不是空白页面 | ✅ P1 重要 |
| **记录删除** | 点击记录卡片可删除单条记录 | ✅ P1 重要 |
| **长按精确设置** | 长按卡片可设置精确的开始时间和时长 | ✅ P1 重要 |

### 3.2 理想周功能

**设计理念**：
> "理想周不是完美计划，是一个温柔的参照系。"不是"我必须做到"，而是"我想往这个方向走"。

**核心逻辑**：
- 用户按天规划理想的一周时间块（如：周一至周五 8:30-18:00 工作，周四 19:00-20:00 运动，20:00-22:00 朋友聚餐）
- 每个时间块关联一个活动类别，用类别颜色显示
- 空白区域 = 未安排的时间 = 用户可以用来安排其他活动的时间
- 通过查看空白处，用户能发现"原来我周六早上是空的，可以安排做家务"

**设置方式**：
- 切换到"理想"视图 → 选择某一天（周一到周日） → 查看该天的时间块排列
- 时间块以 6:00-24:00 的时间轴展示，每个块显示类别名 + 时间范围
- 块之间的空隙显示为虚线框"可安排"区域，标注可用时间段

### 3.3 理想周对比功能（本周视图）

**设计理念**：
> "对比不是为了批判，是为了看见。"
> 理想周对比是一面镜子，不是一把鞭子。

**核心价值**：
- 帮助用户看见"理想的自己"和"真实的自己"之间的差距
- 引导用户思考：是计划不合理，还是执行有问题？
- 最终目标：让理想周越来越接近真实的自己，而不是反过来

**功能定义**：

| 功能 | 说明 | 优先级 |
|---|---|---|
| **双时间轴对比** | 同一天的理想安排和实际记录上下排列对比显示 | ✅ P0 核心 |
| **周几切换** | 周一到周日可切换查看每天的对比情况 | ✅ P0 核心 |
| **三种状态标记** | 符合预期/未完成/额外完成，用不同视觉样式区分 | ✅ P0 核心 |
| **对比摘要** | 显示理想时长、实际时长、符合项数、未完成项数、额外项数 | ✅ P1 重要 |
| **AI 分析建议** | 基于数据分析差距原因，给出调整建议（远期） | 🔮 未来 |
| **一键调整计划** | 基于分析建议，一键调整理想周安排（远期） | 🔮 未来 |

**三种状态定义**：

| 状态 | 定义 | 视觉样式 | 设计意图 |
|---|---|---|---|
| **符合预期** | 理想中有，实际也做了 | 正常颜色显示 | 肯定用户的执行力，也验证计划的合理性 |
| **未完成** | 理想中有，但实际没做 | 半透明/虚线边框 | 不批判，只呈现。让用户自己思考原因 |
| **额外完成** | 理想中没有，但实际做了 | 不同色系（如绿色） | 看见计划外的真实生活，可能是惊喜也可能是黑洞 |

**交互设计**：
- 进入"本周"Tab → 默认显示当天的对比视图
- 顶部有周几切换栏（一/二/三/四/五/六/日）
- 上半部分：理想时间块（浅色/赭石色调）
- 下半部分：实际记录块（深色/主色调）
- 底部：对比摘要卡片

**设计原则**：
1. **不评判，只呈现** — 不用"完成率"、"得分"等评判性语言
2. **颜色不暗示好坏** — 不同状态用不同样式区分，但不用红色暗示"错误"
3. **引导反思，不是说教** — 提供思考方向，不告诉用户"你应该怎样"
4. **计划是活的** — 理想周可以调整，调整不是失败，是更了解自己

**用户价值分层**：

**第一层：看见差距**
- "哦，原来我计划了这么多，但实际只做了一半"
- "原来我每天刷手机的时间比我以为的多很多"

**第二层：发现规律**
- "我发现我周三、周四晚上总是完不成运动计划"
- "因为这两天我总是加班到很晚"
- "所以不是我懒，是晚上运动的计划对我不现实"

**第三层：调整系统**
- "那我把运动挪到中午试试？"
- "或者挪到周末？"
- "理想周不是用来逼自己的，是用来找到适合自己的节奏的"

> **核心理念**：理想周的最终形态，不是一个完美的计划，而是一个和真实的你高度吻合的系统。
> 系统越接近真实的你，它就越稳定，越不需要意志力。
> 这才是"人生系统"的真正含义 —— 不是改造你，是找到你。

### 3.4 习惯生命周期（产品理念，Demo 可简化展示）

| 阶段 | 用户操作 | 系统行为 | 核心价值 |
|---|---|---|---|
| **建立期** | 每天点一下"做了" | 记录行为频率 | 建立习惯 |
| **稳定期** | 只有"没做"才需要点 | 默认"已完成" | 反向记录 |
| **记录期** | 几乎不需要操作 | 自动记录，用于时间复盘 | 知道时间去向 |

### 3.4 不做的事（Scope 边界）

- ❌ AI 对话功能
- ❌ SOP/例程管理
- ❌ 社交/督促功能
- ❌ 番茄钟/日历集成
- ❌ 付费/会员功能
- ❌ 复杂的数据分析
- ❌ 多账户/云端同步
- ❌ 数据导出功能
- ❌ 自定义类别
- ❌ 连续天数统计（避免焦虑）

## 四、用户旅程

### 4.0 核心设计原则

> **"打开即用，点完即走"** —— 把用户的注意力成本降到最低

| 原则 | 说明 | 微习惯理论基础 |
|---|---|---|
| **零跳转记录** | 打开 APP 直接看到记录卡片，无需点击"记录"按钮 | 降低启动阻力 |
| **1 次点击=15 分钟** | 点一下卡片自动记录，连续点击累加 | 微习惯最小单元 |
| **长按精确设置** | 长按卡片可设置精确时间，不打扰日常快速记录 | 渐进式精确 |
| **即时反馈** | 每次操作有明确的视觉、数字反馈 | 让它令人愉悦 |
| **无弹窗设计** | 所有操作在当前页面完成，不打断用户思路 | 减少认知负担 |
| **无压力记录** | 没有连续天数、没有打卡压力、断了也没关系 | 允许弹性和失误 |

### 4.1 旅程一：快速记录时间（核心场景）

**用户目标**：做完一件事，顺手记录花了多久

```
打开 APP → 看到类别卡片 → 点击卡片（可连续点） → 完成记录
     ↓             ↓            ↓                    ↓
  0秒            1秒           1-2秒                0.5秒
```

**详细步骤**：

| 步骤 | 界面状态 | 交互 | 反馈 |
|---|---|---|---|
| 1 | 打开 APP，直接进入"今日"页面 | 无操作 | 页面淡入，类别卡片渐显 |
| 2 | 看到 6 个大尺寸类别卡片 | 无操作 | 卡片有轻微呼吸动效 |
| 3 | 点击"学习"卡片一次 | 点击 | 卡片缩放回弹，数字"+15m"弹出，卡片时间显示"15分钟" |
| 4 | 2 秒内再点击一次 | 点击 | 卡片缩放，数字更新为"+30m"，卡片时间显示"30分钟" |
| 5 | 停顿超过 2 秒 | 无操作 | 系统自动确认，合并为一条 30 分钟的记录，添加到列表 |

**关键交互**：
- **点击合并**：短时间内（2 秒）连续点击同一类别卡片，累加为 n×15 分钟，最终生成一条记录
- 卡片布局：2×3 网格，大尺寸，容易点击
- 点击反馈：缩放 + 颜色变化 + 数字飘出 + 卡片时间实时更新
- 时间感知：记录的时间默认为"当前时间 - N×15 分钟"到"当前时间"

### 4.2 旅程二：精确设置时间

**用户目标**：需要记录非 15 分钟倍数的精确时长

```
打开 APP → 长按类别卡片 → 设置精确时间 → 确认
     ↓             ↓             ↓            ↓
  0秒           0.5秒          2秒          0.5秒
```

| 步骤 | 界面状态 | 交互 | 反馈 |
|---|---|---|---|
| 1 | 看到类别卡片 | 长按"工作"卡片约 500ms | 卡片底部平滑展开时间设置面板 |
| 2 | 面板内容：开始时间选择器 + 时长选择 + "记上"按钮 | 点击时间选择器 | 显示时间滚轮（小时/分钟） |
| 3 | 设置开始时间和时长 | 滑动选择 | 数字实时更新 |
| 4 | 点击"记上"按钮 | 点击 | 面板收起，记录完成 |

### 4.3 旅程三：查看今日记录

**用户目标**：看看今天都做了什么

```
打开 APP → 向下滚动 → 看到记录列表
     ↓         ↓             ↓
  0秒        1秒            2秒
```

| 步骤 | 界面状态 | 交互 | 反馈 |
|---|---|---|---|
| 1 | 打开 APP，看到顶部类别卡片和底部记录列表 | 无操作 | 记录列表自动展开 |
| 2 | 记录列表按时间倒序排列 | 无操作 | 最新记录有淡入动画 |
| 3 | （可选）点击一条记录 | 点击 | 显示"删除"按钮 |
| 4 | （可选）点击"删除"按钮 | 点击 | 记录滑出消失，总时长递减 |

### 4.4 旅程四：查看本周对比

**用户目标**：了解本周每天的实际行为与理想计划的差距

```
点击"本周"Tab → 选择某一天 → 看到理想 vs 实际对比 → 发现差距
     ↓              ↓                ↓                  ↓
  0秒           0.3秒          1秒                 2秒
```

| 步骤 | 界面状态 | 交互 | 反馈 |
|---|---|---|---|
| 1 | 点击底部"本周"Tab | 点击 | 页面切换到本周对比视图 |
| 2 | 看到周一到周日的日期标签 | 无操作 | 当前选中天高亮 |
| 3 | 点击"周六"标签 | 点击 | 切换到周六的对比视图 |
| 4 | 看到时间轴：理想块（赭石色）和实际块（深色）并排显示 | 无操作 | 理想计划 vs 实际执行一目了然 |
| 5 | 发现理想中周六早上安排了"整理家务"，但实际记录的是"玩游戏" | 认知 | 意识到计划与现实的差距 |
| 6 | 底部显示对比摘要：理想时长 vs 实际时长，符合预期/未完成/额外完成的项目数 | 无操作 | 快速掌握当天整体情况 |

**对比视图设计**：
- **理想块**：赭石色背景 + 左边框，显示计划的活动和时间
- **实际块**：深色背景 + 左边框，显示真实记录的活动和时间
- **缺失块**：理想计划中有但实际未执行的活动，虚线边框标记
- **额外块**：理想计划中没有但实际执行了的活动，绿色标记

### 4.5 旅程五：理想周排程

**用户目标**：规划理想的一周，看到哪些时间还可以安排

```
     ↓              ↓              ↓                ↓

```

| 步骤 | 界面状态 | 交互 | 反馈 |
|---|---|---|---|
| 1 | 点击底部"理想"Tab | 点击 | 页面切换到理想周 |
| 2 | 看到周一到周日的日期标签 | 无操作 | 当前选中天高亮 |
| 3 | 点击"周四"标签 | 点击 | 切换到周四的时间块视图 |
| 4 | 看到周四的时间排列：8:30-18:00 工作、19:00-20:00 运动、20:00-22:00 聚餐 | 无操作 | 每个块按类别颜色显示 |

**用户目标**：第一次使用，快速了解怎么用

## 五、页面结构

### 5.1 信息架构

```
TapTime App

├── Today（今日）← 主页面（打开即进入）
│   ├── 顶部：日期 + 总时长
│   ├── 类别卡片区：6 个大尺寸记录卡片（2×3 网格）
│   └── 记录列表：今日记录（卡片下方，可滚动）
│
├── Week（本周对比）
│   ├── 顶部：标题 + 日期范围
│   ├── 日期标签：[一] [二] [三] [四] [五] [六] [日]
│   ├── 图例：理想（赭石色）/ 实际（深色）
│   ├── 对比时间轴：理想块 vs 实际块并排显示
│   │   ├── 理想块：计划的活动，赭石色
│   │   ├── 实际块：真实记录，深色
│   │   ├── 缺失块：理想中有但未执行，虚线标记
│   │   └── 额外块：理想中没有但执行了，绿色标记
│   └── 对比摘要：理想时长 vs 实际时长，符合/未完成/额外完成数
│
└── Ideal（理想周）
    ├── 顶部：标题 + "规划你的一周"
    ├── 日期标签：[一] [二] [三] [四] [五] [六] [日]
    └── 时间轴：6:00-24:00 的当日时间块排列
        ├── 已安排块：类别颜色 + 名称 + 时间范围
        └── 空白块：虚线框"可安排" + 可用时间段
```

### 5.2 底部导航

- 三个 Tab："今日" / "本周" / "理想"
- 当前选中：黑色文字 + 底部指示线
- 未选中：灰色文字
- 切换方式：点击 Tab

## 六、交互规范

### 6.1 核心交互

**时间块卡片交互（点击合并机制）**：

| 操作 | 反馈 | 说明 |
|---|---|---|
| 点击 1 次 | 卡片缩放 + "+15m" 飘出 + 卡片显示"15分钟" | 开始一次记录 |
| 2 秒内再点击 | "+30m" 飘出 + 卡片显示"30分钟" | 累加为同一记录 |
| 2 秒内第 3 次点击 | "+45m" 飘出 + 卡片显示"45分钟" | 继续累加 |
| 停顿超过 2 秒 | 系统自动确认，生成一条 N×15 分钟的记录 | 合并完成 |
| 长按 500ms | 展开精确设置面板 | 设置精确时间 |

**关键设计**：短时间内（2 秒窗口）连续点击同一类别，不会生成多条记录，而是合并为一条 n×15 分钟的记录。这避免了记录列表被碎片化的 15 分钟条目塞满。

**手势支持**：

| 手势 | 作用 |
|---|---|
| **点击** | 记录时间块 / 切换 Tab / 选择理想周日期 |
| **长按** | 触发精确设置面板 |
| **下拉** | 刷新/重新加载（可选） |

### 6.2 类别配置

| 类别 ID | 名称 | 图标 | 颜色 |
|---|---|---|---|
| work | 工作 | 💼 | 赭石色 |
| reading | 阅读 | 📖 | 雾霾蓝 |
| exercise | 运动 | 🏃 | 苔藓绿 |
| rest | 休息 | ☕ | 卡其棕 |
| eating | 吃饭 | 🍽 | 奶茶色 |
| gaming | 游戏 | 🎮 | 紫罗兰 |

### 6.3 数据结构

**记录数据格式**：

```json
{
  "records": {
    "2026-07-09": [
      {
        "id": "uuid-1",
        "category": "learn",
        "duration": 30,
        "time": "09:30 — 10:00"
      }
    ]
  },
  "idealWeek": {
    "0": [
      { "category": "work", "startHour": 8, "startMin": 30, "duration": 570 },
      { "category": "cook", "startHour": 18, "startMin": 0, "duration": 60 }
    ],
    "3": [
      { "category": "work", "startHour": 8, "startMin": 30, "duration": 570 },
      { "category": "fitness", "startHour": 19, "startMin": 0, "duration": 60 },
      { "category": "rest", "startHour": 20, "startMin": 0, "duration": 120 }
    ]
  }
}
```

**数据字段说明**：
- `records`：按日期分组的记录对象，key 为日期（YYYY-MM-DD）
- `duration`：时长（分钟），支持值：15、30、45、60、90、120 等 15 的倍数
- `time`：记录的时间范围（HH:MM — HH:MM）
- `idealWeek`：按天（0=周一 ... 6=周日）的理想时间块数组
- 每个理想块包含 `category`、`startHour`、`startMin`、`duration`（分钟）

### 6.4 数据持久化

- 使用 localStorage 存储
- 键名：`taptime_records`
- 页面加载时读取，每次操作后保存
- 首次打开无数据时，使用预设数据初始化

## 七、设计方向

### 7.1 整体风格

**Editorial Magazine 极简风** —— 参考欧美设计师品牌官网和艺术类 App 的设计语言。

**关键词**：极简、留白、有温度、仪式感、克制、高级

**感觉**：
- 像一本精致的杂志，不是一个工具 App
- 像手写日记一样有温度，不是冷冰冰的数据面板
- 像设计师品牌官网一样克制，不是功能堆砌的产品

### 7.2 设计原则（不限制具体实现）

- **大量留白**：不要让界面拥挤
- **有温度的极简**：不是冷冰冰的黑白，要有温度感
- **类别颜色点缀**：用类别颜色作为视觉节奏，但不要过度使用
- **数字突出**：总时长等关键数字要显眼
- **动效克制**：每次操作有反馈，但不要太花哨

### 7.3 竞品参考

| 产品 | 学习的点 | 避免的点 |
|---|---|---|
| **Nosh** | 日记本+贴纸视觉、极简交互、仪式感 | - |
| **HabitDone** | 简洁克制、反向打卡逻辑 | 功能太简单 |
| **MarkTimes** | 时间块记录方式、设计美感 | 易用性差 |
| **滴答清单** | - | 功能太多太杂、信息密度过高 |

## 九、品牌调性与文案风格

### 9.1 品牌调性

- **温和但坚定**：不说教，不鸡血，像朋友在建议
- **克制不炫耀**：好的设计是看不见的设计
- **有温度**：不是冷冰冰的工具，是陪伴成长的伙伴
- **真实不虚假**：不承诺"30 天改变人生"，只说"慢慢来，比较快"

### 9.2 文案风格

- 简短，不啰嗦
- 口语化，不官方
- 有画面感，不抽象
- 像和朋友聊天，不是产品说明书

**好的例子**：
- "轻点一下，记录时间"
- "知道自己怎么过的，成为想成为的人"
- "今天还没有记录，点击下方按钮开始"
